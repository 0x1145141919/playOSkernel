## 内核内存子系统初始化时序（kernel.elf 阶段）

本部分聚焦于内核启动后（`kernel_start` 函数开始）的内存子系统初始化顺序。整个过程严格依赖前序阶段（bootloader + init.elf）传递的 `init_to_kernel_info` 结构体，该结构体已提供物理内存视图、预分配的 VM 区间、页表根地址等关键信息。

---

### 1. 初始接收与基础环境准备

**函数入口**：`kernel_start(init_to_kernel_info* transfer)`

#### 1.1 解析传递信息
- **操作**：从 `transfer` 中提取关键数据：
  - 物理内存段数组 `phymem_segments` 及其数量
  - 预定义的 VM 区间数组 `loaded_VM_intervals` 及其数量
  - 透传设备信息（如 GOP 帧缓冲）
  - 内核页表根物理地址 `kmmu_root_table`
  - UEFI 系统表指针 `gST_ptr`
  - 符号表文件信息
- **依赖**：`init_to_kernel_info` 结构体必须在物理内存中有效且可访问。
- **产出**：内核获得物理内存视图、VM 区间列表等基础信息。

#### 1.2 基础图形与控制台初始化
- **操作**：
  - `GfxPrim::Init()`：初始化图形输出基元，获取帧缓冲区信息。
  - `textconsole_GoP::Init()` 与 `textconsole_GoP::Clear()`：初始化文本控制台，清屏。
  - `serial_init_stage1()` 与 `bsp_kout.Init()`：初始化串口和内核输出系统，使 `bsp_kout` 可工作。
- **依赖**：透传设备中必须包含 GOP 设备信息。
- **产出**：早期调试输出通道就绪。

#### 1.3 符号表与第一个堆初始化
- **操作**：
  - `ksymmanager::Init(ksymbols, transfer->ksymbols_file_size)`：加载内核符号表，用于调试和回溯。
  - `kpoolmemmgr_t::Init(first_heap, first_heap_bitmap)`：初始化第一个堆（线性堆），该堆使用预先分配并映射好的物理内存，提供早期 `new`/`delete` 能力。
- **依赖**：`transfer` 中必须包含 `VM_ID_FIRST_HEAP` 和 `VM_ID_FIRST_HEAP_BITMAP` 区间。
- **产出**：内核可以动态分配内存（早期阶段）。

#### 1.4 数据复制到内核空间
- **操作**：
  - 将 `transfer->loaded_VM_intervals` 和 `transfer->memory_map` 复制到由 `new` 分配的内核内存中。
  - 保存这些数组的指针和长度到全局变量（`VM_intervals`, `phymem_segments` 等）。
- **依赖**：`kpoolmemmgr_t::Init` 已经完成，`new` 可用。
- **产出**：内核拥有了物理内存视图和 VM 区间的独立副本。

---

### 2. 核心内存管理器基础初始化

函数：all_pages_arr::Init(transfer)

操作：

    从 transfer->loaded_VM_intervals 中查找 VM_ID_MEM_MAP 区间，获取页框数组的虚拟地址和大小。

    根据 transfer->memory_map（物理内存段）和 transfer->loaded_VM_intervals（已分配的 VM 区间）两类信息，初始化页框数组 mem_map。

    初始化逻辑：

        基于物理内存段：遍历 phymem_segments，根据段类型（如 freeSystemRam、EFI_ACPI_MEMORY_NVS 等）标记对应物理页的状态（free、acpi_tables、reserved 等）和可分配标志。

        基于 VM 区间：遍历 loaded_VM_intervals，将已被内核占用的物理页（如堆、栈、日志缓冲等）标记为 kernel_persistent（不可分配）。图形缓冲区单独标记为 mmio。

        特殊区域标记：将 init_to_kernel_info 结构体自身和初始内核页表占用的物理内存也标记为 kernel_persistent。

    以及从中获取最大物理内存地址，将其填入PhyAddrAccessor::BASIC_DESC的特殊字段，供其它子系统使用

依赖：

    phymem_segments 已正确建立，VM 区间列表已准备就绪。

    页框数组的物理内存已在 init.elf 阶段分配并映射到虚拟地址，此处直接使用该虚拟地址。

产出：

    全局页框数组 mem_map 完全初始化，每个物理页的 head.type（页面类型）、page_flags.is_allocateble（是否可分配）等字段被正确设置。

    PhyAddrAccessor::BASIC_DESC.SEG_SIZE_ONLY_UES_IN_BASIC_SEG 被设置为页框数组的大小（用于物理地址访问器的边界检查）。

关键点：

    页框数组的大小由 VM_ID_MEM_MAP 区间的大小决定，init.elf 必须保证其足够容纳所有物理页对应的 page 结构。

    该函数不进行物理页分配，仅标记状态，为后续物理页分配器（FreePagesAllocator）提供决策依据。。

#### 2.2 物理页分配器（Buddy System）初始化（第一阶段）
- **函数**：`FreePagesAllocator::Init()`
- **操作**：
  - 扫描所有空闲物理内存段，将大块连续内存切分为不同 order 的 Buddy Control Block (BCB)。
  - 选择最小的可用 order 作为第一个 BCB（`first_BCB`），并对其进行初始化。
  - 该阶段仅建立了单块 BCB（`first_BCB`），用于后续早期分配。
- **依赖**：页框数组已初始化，物理内存视图已明确。
- **产出**：物理页分配器可用（但功能受限，仅通过 `first_BCB` 分配）。

2.3 内核页表管理器初始化

    函数：KspacePageTable::Init(loaded_VM_interval* kspace_up_layer)

    操作：

        保存页表区域信息：

            将 kspace_up_layer 的物理地址保存到 kspace_uppdpt_phyaddr。

            将虚拟地址保存到全局指针 kspaceUPpdpt，供后续页表操作使用。

        创建内核空间 VM 管理器：

            kspace_vm_table = new kspace_vm_table_t()：创建红黑树，用于管理内核虚拟地址空间的分配情况。

        建立预分配 VM 区间的映射：

            遍历 VM_intervals（全局变量，已从 transfer 复制），筛选出虚拟地址在 PAGELV4_KSPACE_BASE（高半区基址，通常为 0xffff800000000000）以上的区间。

            对每个符合条件的区间：

                调用 enable_VMentry(interval) 建立页表映射（物理地址 → 虚拟地址）。

                将区间信息插入 kspace_vm_table 红黑树，标记该虚拟地址范围已被占用。

        为物理地址访问器分配虚拟地址空间：

            从 kspace_vm_table->alloc_available_space() 分配一块与 PhyAddrAccessor::BASIC_DESC.SEG_SIZE_ONLY_UES_IN_BASIC_SEG 大小相等的虚拟地址空间（该大小已在 all_pages_arr::Init 中设置为页框数组大小）。

            调用 enable_VMentry() 建立该虚拟地址空间到物理地址 0 的映射（恒等映射），权限为 PG_RW（内核可读写）。

            将映射信息保存到 PhyAddrAccessor::BASIC_DESC，并插入 kspace_vm_table 红黑树。

        输出调试信息：调用 kspace_vm_table->all_node_print() 打印所有已插入的 VM 区间。

    依赖：

        VM_intervals 全局数组已从 transfer->loaded_VM_intervals 复制完成。

        PhyAddrAccessor::BASIC_DESC.SEG_SIZE_ONLY_UES_IN_BASIC_SEG 已在 all_pages_arr::Init 中设置。

        all_pages_arr 已初始化，页框数组可用（间接依赖）。

        页表操作底层函数（如 _4lv_pte_4KB_entries_set）可用。

    产出：

        内核页表管理器（KspacePageTable 静态接口）初始化完成，可进行页表映射操作。

        kspace_vm_table 红黑树包含：

            所有高半区预分配 VM 区间（堆、栈、日志缓冲、页框数组等）。

            为 PhyAddrAccessor 分配的虚拟地址区间（用于访问任意物理地址）。

        PhyAddrAccessor::BASIC_DESC 被正确配置，使 PhyAddrAccessor 能够通过虚拟地址访问任意物理地址（用于后续 ACPI 表读取等操作）。

        所有预分配 VM 区间已建立页表映射，内核可访问这些区域。

    关键点：

        该函数在切换到内核页表之前执行，此时仍使用 init.elf 建立的页表，但通过 enable_VMentry 直接修改当前页表，添加新的映射。

        enable_VMentry 会调用底层页表操作，根据区间大小和地址对齐自动选择 4KB、2MB 或 1GB 大页映射。

        kspace_vm_table 红黑树不仅用于记录已分配区间，还用于后续虚拟地址分配时的冲突检测（通过 alloc_available_space）。

        为 PhyAddrAccessor 分配虚拟地址空间是关键步骤，使得内核在切换到最终页表后仍能通过该虚拟地址区间访问任意物理地址（如读取 ACPI 表、设备寄存器等）。

#### 2.4 内核虚拟地址空间管理器初始化
- **函数**：`AddressSpace::second_stage_init()`
- **操作**：
  - 创建 `AddressSpace` 对象 `gKernelSpace`，用于管理整个内核的虚拟地址空间。
  - 该函数会接管 init.elf 已建立的部分页表，并准备好后续映射接口。
- **依赖**：`KspacePageTable::Init` 已完成。
- **产出**：内核空间管理器可用。

#### 2.5 建立完整内核空间映射
- **操作**：通过 `gKernelSpace->enable_VM_desc` 循环建立映射：
  1. **恒等映射**：遍历 `phymem_segments`，将所有物理内存段映射到相同虚拟地址（低端），对 MMIO 等区域使用 UC 缓存策略。
  2. **预分配 VM 区间映射**：遍历 `VM_intervals`，目的是给低位trampoline进行映射，以便其它核心切换到长模式不会三重错误
- **依赖**：`gKernelSpace` 可用，页表操作功能就绪。
- **产出**：内核空间完成对所有物理内存和预分配区域的映射，此时内核可以访问所有物理内存和预定义区域。

#### 2.6 切换到内核页表
- **操作**：`gKernelSpace->unsafe_load_pml4_to_cr3(KERNEL_SPACE_PCID)`
  - 将内核页表的根物理地址加载到 `cr3` 寄存器，启用最终的页表。
- **依赖**：所有映射已建立。
- **产出**：CPU 使用内核页表，内存视图切换完成。

### 3. 内存管理器高级初始化

#### 3.1 物理页分配器第二阶段（多 BCB 建立）
- **函数**：`FreePagesAllocator::second_stage(strategy)`
- **操作**：
  - 根据策略（通常为 `INIT_STRATEGY_MATCH_THREAD`）将剩余的大块物理内存划分为多个 BCB。
  - 为每个 BCB 分配内存（从 `first_BCB` 中分配）并初始化。
  - 构建 BCB 数组（`mainBCBS` 和 `vice_BCBS`），并准备每 CPU 的统计信息。
- **依赖**：`first_BCB` 可用（通过第一阶段），多核信息可用（通过 ACPI APIC 表，此前已初始化 `gAnalyzer`）。
- **产出**：完整的 Buddy System 分配器就绪，支持多 BCB 并发分配，提高性能。

#### 3.2 内核堆管理器多堆启用
- **函数**：`kpoolmemmgr_t::multi_heap_enable()`
- **操作**：
  - 为每个 CPU 核心预留多块堆内存（HCB），并预先分配虚拟地址空间。
  - 初始化全局 HCB 数组，供后续每 CPU 堆使用。
- **依赖**：物理页分配器已完全就绪（第二阶段完成），虚拟地址空间管理器可用。
- **产出**：支持每 CPU 堆，减少锁竞争，提高内核内存分配效率。

#### 3.3 内核堆管理器每 CPU 堆分配
- **时机**：当应用线程或中断处理程序首次调用 `kalloc` 时，会自动为当前 CPU 分配一个堆。
- **操作**：`kpoolmemmgr_t::alloc_heap(idx)` 从预先预留的虚拟地址空间和物理页分配器中为指定 CPU 创建新的堆。
- **依赖**：`multi_heap_enable` 已完成，物理页分配器可用。
- **产出**：每个 CPU 拥有独立的堆，实现无锁或低锁开销分配。

---

### 4. 内存子系统初始化完成标志

- **全局状态**：`GlobalKernelStatus` 在 `kernel_start` 中依次更新：
  - `EARLY_BOOT` → `MM_READY`（切换页表后）→ `SCHEDUL_READY`（最终）。
- **可用的内存接口**：
  - 物理页分配：`FreePagesAllocator::alloc` / `free`
  - 内核对象分配：`new` / `delete`（基于 `kpoolmemmgr_t`）
  - 虚拟地址映射：`gKernelSpace->enable_VM_desc` / `disable_VM_desc`

---

### 5. 时序依赖关系总结

| 序号 | 初始化函数 | 依赖条件 | 产出 |
|------|------------|----------|------|
| 1 | `kpoolmemmgr_t::Init` | `transfer` 中堆区间存在 | 早期堆可用 |
| 2 | `all_pages_arr::Init` | `phymem_segments` 已复制 | 页框数组就绪 |
| 3 | `FreePagesAllocator::Init` | 页框数组就绪 | 单一 BCB 分配器 |
| 4 | `KspacePageTable::Init` | `kspaceUPpdpt` 区间已映射 | 页表操作接口 |
| 5 | `AddressSpace::second_stage_init` | 页表操作接口 | 内核空间管理器 |
| 6 | 建立恒等映射与 VM 区间映射 | 内核空间管理器 | 完整内核映射 |
| 7 | 切换到内核页表 | 完整映射 | 正式页表生效 |
| 8 | `FreePagesAllocator::second_stage` | 多核信息，第一阶段分配器 | 多 BCB 分配器 |
| 9 | `kpoolmemmgr_t::multi_heap_enable` | 多 BCB 分配器，虚拟地址空间 | 多堆框架 |
| 10 | 按需分配每 CPU 堆 | `multi_heap_enable` 完成 | 高性能每 CPU 堆 |

---

### 6. 关键设计要点

- **分阶段初始化**：内存模块分为“基础可用”和“高性能优化”两个阶段，确保早期代码能正常运行，后续再逐步启用高级特性。
- **依赖传递**：内核启动时所有关键信息均由 `init_to_kernel_info` 传入，避免了内核早期依赖外部服务。
- **页表隔离**：内核页表与用户空间页表分离（通过 `KERNEL_SPACE_PCID` 标识），为后续进程管理奠定基础。
- **堆分配策略**：通过“第一个堆（线性堆）+ 每 CPU 堆（Buddy 堆）”的分层设计，平衡了简单性和性能。

以上时序和依赖关系确保了内核在启动过程中能够安全、高效地管理内存，为后续模块（如调度器、中断系统）提供可靠的内存服务。
